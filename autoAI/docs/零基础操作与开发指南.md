# autoAI 零基础操作与开发指南

这份文档是给“不会编程也能用”的版本。你只需要会复制命令、看结果、按步骤执行。

---

## 1. 你现在得到了什么能力

项目已经具备这条闭环：

1. 发送消息到 AI 网站（ChatGPT/Claude 等）  
2. 自动记录失败（结构化字段）  
3. 自动导出失败快照（JSON + 事件流）  
4. 在 CI 中按门禁规则自动判定通过/失败

其中你最常用的是这 2 个产物：

- `artifacts/failure-snapshot.json`（给人看 + 给 CI 看）
- `artifacts/failure-snapshot.events.ndjson`（逐行事件流，适合统计平台）

---

## 2. 最短可执行操作（每天就按这个来）

在项目根目录执行：

```bash
npm run regression:loop
```

如果看到类似输出：

- `failure snapshot written: .../artifacts/failure-snapshot.json`
- `failure event stream written: .../artifacts/failure-snapshot.events.ndjson`
- `regression loop passed`

说明这轮回归通过。

---

## 3. 如何理解“通过/失败”

门禁由环境变量控制（你可以把它理解为“自动验收标准”）：

- `AUTOAI_CI_MAX_FAILURES`：允许最多失败事件数（默认 `0`）
- `AUTOAI_CI_MAX_UNKNOWN`：允许最多 UNKNOWN 错误数（默认 `0`）

例子：允许最多 3 个失败，且 UNKNOWN 最多 1 个

```bash
AUTOAI_CI_MAX_FAILURES=3 AUTOAI_CI_MAX_UNKNOWN=1 npm run regression:loop
```

如果超过阈值，脚本会非 0 退出，CI 会自动失败。

## 4. 背后原理（用业务语言解释）

你可以把系统想成 3 层：

1. **日志标准化层**  
   任何失败都尽量被记录成统一字段：  
   `errorCode/stage/sendSeq/path/siteId/retryable`

2. **自动定位层**  
   系统会做规则判定（例如：是超时、是注入失败、还是 CDP 页面绑定问题）。

3. **有限自动修复层**  
   对“低风险且可重试”的问题自动尝试一次修复（有次数和退避护栏）。

为什么 `sendSeq` 很重要：  
每次发送都有唯一 ID，能把“本轮发送”的所有日志串起来，避免和历史日志混淆。

## 5. failure-snapshot 结构说明（你重点看这个）

### 5.1 `failure-snapshot.json`

- `schemaVersion`: 当前格式版本
- `events[]`: 结构化失败事件数组
- `summary`: 聚合统计（按错误码、按阶段）
- `gate`: 门禁判定结果（是否通过）

单条事件核心字段：

- `errorCode`: 错误码（如 `TIMEOUT_EMPTY_BODY`）
- `stage`: 阶段（如 `inject/network/settle`）
- `sendSeq`: 本次发送唯一序号
- `path`: 执行路径（`playwright` / `legacy`）
- `siteId`: 站点 ID
- `retryable`: 是否建议自动重试

### 5.2 `failure-snapshot.events.ndjson`

每行一个 JSON 事件，适合：

- 导入日志平台（ELK、Datadog、ClickHouse 等）
- 做趋势图（每天失败数、错误码 Top N）
- 做自动告警（例如 UNKNOWN 突增）

## 6. 你遇到问题时怎么处理（无代码版）

### 情况 A：门禁失败

1. 打开 `artifacts/failure-snapshot.json`
2. 先看 `gate`
3. 再看 `summary.byErrorCode`
4. 再看 `events` 中 `errorCode` 最多的那类

### 情况 B：你看不懂技术细节

把这三个文件发给 AI（或同事）：

- `artifacts/failure-snapshot.json`
- `artifacts/failure-snapshot.events.ndjson`
- `artifacts/regression-loop.log`

并附一句话：  
“请按 errorCode 排序给出修复优先级，并给出最小改动方案。”

## 7. 以后怎么继续开发（路线图）

你可以按这个顺序推进，风险最低：

1. **先完善错误码覆盖率**  
   目标：`UNKNOWN` 持续下降。

2. **再优化自动修复策略**  
   只针对高频、低风险错误码开修复动作。

3. **再做站点灰度**  
   先 ChatGPT，再 Claude，再扩到其它站点。

4. **最后做周报指标**  
   每周看 4 个数：成功率、超时率、自动修复触发率、自动修复成功率。

## 8. 给未来自己的“最小开发说明”

如果你后续要新增一个错误码（比如 `SSE_SCHEMA_CHANGED`），按这 4 步：

1. 在 `src/main/automation/failure-codes.ts` 增加错误码
2. 在 `src/main/automation/failure-classifier.ts` 增加判定规则
3. 在主链路调用 `recordChatFailure()` 时填写 `code/stage/retryable/sendSeq`
4. 运行 `npm run regression:loop`，确认快照中有新错误码

## 9. 常见问答（非技术）

- **Q: 我不想看代码，只想知道今天能不能用？**  
  A: 跑 `npm run regression:loop`，看是否通过即可。

- **Q: 为什么要两种文件（json + ndjson）？**  
  A: JSON 方便人读，NDJSON 方便机器统计。

- **Q: 我怎么知道修复是否真的有效？**  
  A: 关键看同一 `errorCode` 的数量是否下降，`gate.passed` 是否稳定为 `true`。
