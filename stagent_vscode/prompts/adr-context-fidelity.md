资深做法补充（借鉴 grill-with-docs 的 ADR-FORMAT / CONTEXT-FORMAT，提升决策与词汇沉淀质量）：

【ADR（docs/adr/，顺序编号 0001-slug.md，懒创建：首个 ADR 需要时才建目录）】
- 仅当三条**同时**成立才产 ADR，否则不写（易反悔的别记，反正会改；不令人意外的没人会问；无真备选的没什么可记）：
  1) 难以反悔——以后改主意代价可观；
  2) 无背景会令人困惑——未来读者看代码会问「为啥这么做」；
  3) 真实取舍的结果——确有备选，为具体理由选了其一。
- 模板极简：`# 标题` + 1–3 句（背景 / 决定了什么 / 为什么），一段话即可；价值在记录「做了某决定」与「为什么」，不在填章节。可选 Status / Considered Options / Consequences 仅在确有价值时加。
- 适合写 ADR 的内容：架构形态、上下文间集成模式（事件 vs 同步 HTTP）、带锁定的技术选型（DB/消息总线/认证/部署目标）、边界与范围（含明确的「不做」）、刻意偏离常规路径（注明原因，免得后人「修正」）、代码看不出的约束（合规/性能 SLA）、非显然的被否方案。

【CONTEXT.md / 项目词汇表（仓库根，懒创建；多上下文用 CONTEXT-MAP.md）】
- 有主见：同一概念有多词时选最佳，其余列 `_Avoid_`。
- 定义紧凑：1–2 句，定义它「是什么」而非「做什么」。
- 只收**本项目特有**术语；通用编程概念（超时、错误类型、工具模式）不收——加术语前先问：这是本上下文独有概念还是通用编程概念？只有前者才收。
- 决策阶段与 grill 须**复用并扩充**该词汇表，使阶段/接口/测试命名与项目语言一致；遵守相关 ADR，避免重复 litigate 已决架构。
